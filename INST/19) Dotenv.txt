


Работа с переменными окружения — это стандарт безопасности. 
Это позволяет разделять настройки для разных сред (разработка, тест, продакшен) и не "светить" секретные ключи 
в публичных репозиториях.

1. Установка плагина
Устанавливаем пакет, который подружит Webpack с .env файлами:

npm install dotenv-webpack --save-dev


//////////////////////////////////////////////
2. Создание файлов окружения
В корне проекта (рядом с package.json) создай файл .env. Здесь будут храниться твои настройки:


Важно: Сразу создай файл .gitignore (если его нет) и добавь туда .env. Эти данные не должны попасть на GitHub!


//////////////////////////////////////////////

3. Настройка в webpack.config.js

import Dotenv from 'dotenv-webpack';


plugins: [
    new Dotenv({
      path: './.env', // Путь к файлу (по умолчанию он и так такой)
      safe: true,     // Можно создать .env.example, чтобы проверять наличие всех переменных
      systemvars: true // Позволяет считывать переменные из самой ОС (полезно для CI/CD)
    }),
]


Нюанс для TypeScript
TypeScript "из коробки" не знает, что у объекта process.env есть твои кастомные поля. 
Чтобы он не ругался, создай в папке src файл env.d.ts:

declare namespace NodeJS {
  interface ProcessEnv {
    API_URL: string;
    SENTRY_KEY: string;
    DEBUG: string;
  }
}


в корне проекта нужно создать файл .env
в корне проекта нужно создать файл .env.example

API_URL='vxcvxcvxv'
SENTRY_KEY='xcvxcvxcv'
DEBUG='xcvxcvxcv'


Последний важный совет по .env
Сейчас, когда всё работает, запомни золотое правило:
.env — никогда не добавляй в Git (там твои секреты).
.env.example — всегда добавляй в Git (это инструкция для других).
Если ты захочешь добавить новую переменную, например DATABASE_URL, сначала добавь её в .env.example, а потом в свой .env. Твой "строгий" Webpack проследит, чтобы ты ничего не забыл.
